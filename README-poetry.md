# 诗歌虫洞实证 — DeepSeek V4 Flash（MoE）版

> 本分支（`silicon-poets-experiment`）在 DeepSeek V4 Flash 上重做"古诗语义正交"实验。
> 配套公众号：121 期（Llama 版）+ 214 期（本 MoE 版）。
> 原 dense 模型实验代码见 [silicon-poets-experiment](https://github.com/lmxxf/silicon-poets-experiment)。

## 这是在做什么

EmrysRyu《硅基诗学》提出：好诗的字和字之间在高维语义空间里**互相正交**，桥接字同时连接两个语义域。

121 期已在 Llama-3.3-70B（dense）上用余弦相似度 + EID + SAE 验证过。本分支换到 **DeepSeek V4 Flash（280B，MoE，256 专家选 6 + 1 共享）**，目的有二：

1. **稳健性**：换个架构，"字和字正交"还成立吗？
2. **新视角**：MoE 的专家路由是明摆着的——每个字被派给哪些专家可以直接 hook 出来。dense 模型只能用余弦间接推正交，MoE 能**直接看见**正交（正交的字 → 派给不重叠的专家）。

SAE 实验无法复现（没有 V4 的 SAE），故砍掉，本版只做余弦 + EID + 专家路由。

## 运行环境

本实验复用本仓库的双机平台（流式加载 280B、Triton kernel、TP=2），脚本依赖 `model.py` / `kernel_sm121.py` / `weight_loader.py`，**必须在本目录、容器内跑**，不能换目录。

## 怎么跑

```bash
# 1. 跑实验（双机 TP=2，加载约 7 分钟，5 组诗一次跑完）
SCRIPT=probe_all.py ./run_dual_node.sh
# 产出 probe_all.pt（已 gitignore，不进 git）

# 2. 离线分析（纯 CPU，宿主机直接跑）
python3 probe_all_analyze.py
```

`probe_all.py` 一次 prefill 同时抽两样东西：
- **hidden states**（layer 14/28/42 的 4 路 Hyper-Connection 残差流）→ 算余弦正交 + EID
- **MoE 路由**（每层 Gate 的 top-6 专家 id）→ 算专家正交 + 桥接枢纽 + 哈希/分数对比

输入是**裸文本**（不套 chat template），复现原实验"纯语言结构"测量口径。

## 文件

| 文件 | 说明 |
|------|------|
| `probe_all.py` | 主实验脚本（5 组诗，一次加载全跑），在平台容器内运行 |
| `probe_all_analyze.py` | 离线分析（余弦正交 + EID + 专家路由三条线），含字节碎片合并解码 |
| `probe_all.pt` | 实验产出的激活值 + 路由数据（gitignore，需自己跑出来） |

中间探针版（`probe_poem.py` / `probe_router.py`）是迭代过程稿，已被 `probe_all.py` 合并取代。

## 结论速览

### 1. 余弦正交 — 跨架构复现 ✅

最后一层附近（layer 28）非对角线高相似度 token 对：诗全部 ≤1，白话普遍更多。**"字和字正交"在 MoE 架构上同样成立**，是语言结构的属性，不是某台模型的脾气。

### 2. EID — 部分复现，弱于 dense ⚠️

归一化 EID 比值 1.1~2.2x，全部 >1，但没有 Llama 上 2~3x 那么干净。**余弦正交是稳健信号（跨架构都在），EID 是脆弱信号（对模型维度敏感，换架构打折）。** 老实记录。

### 3. 专家路由 — 正交的"另一把显微镜" 🔥

- **桥接字坐中间**：5 个桥接字（知/直/闻/暮/齐）和句中各字普遍 0.33~0.71 专家重叠，没有掉到 0——同时搭着两侧语义域。其中「闻」（《鹿柴》）和视觉域「空/山/不见」都是 0.50 重叠，与原实验 SAE 测出的"闻是桥接字"**同一结论，两把不同显微镜**。
- **哈希→分数相变**：前 3 层哈希路由（专家由 token id 定死，跟语义无关），第 3 层起分数路由（按语义选专家）。浅层每个字"各进各家"（~50 个专家全散开），深层"全班抱团"（~18 个 + 一批共享专家）。
- **"文言文专家"**：L3 上专家 46 / 69 / 143 / 175 / 186 在 5 首诗里全部出现——不管谁写的、写什么题材，只要是"一句古诗"就点亮这批专家。

## 一个有趣的小坑：「鹜」不是一个字

跑"落霞与孤鹜齐飞"时，「鹜」在模型里占了**两个位置**。原因：BPE 分词对生僻字按字节拆分。「鹜」太生僻，没有独立 token，被拆成 2 个字节 token（`e9b9` + `9c`，拼起来正是「鹜」的 UTF-8）。单独解码每个字节 token 会显示成乱码 `�`——这不是 bug，是分词机制，模型数值全对。`probe_all_analyze.py` 已加字节碎片合并，正常显示「鹜」。
